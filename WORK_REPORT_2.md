## 1. Ошибки компиляции (`Assembly-CSharp-Editor`)

**Что это за ошибка.** `Failed to find entry-points: Mono.Cecil.AssemblyResolutionException:
Failed to resolve assembly 'Assembly-CSharp-Editor'` — это сообщение **Burst-компилятора**
на этапе сборки/сканирования сборок. В 99% случаев это **следствие**, а не причина: либо
где-то есть более ранняя ошибка компиляции C# (тогда `Assembly-CSharp-Editor.dll` просто не
собирается, и Burst не может её найти), либо повреждён кэш `Library` после недавнего
добавления пакетов (вы как раз только что добавили `ROS-TCP-Connector` и `URDF-Importer`).

**Проверка, которую я провёл (headless):**
- Все 9 C#-скриптов проекта ссылочно консистентны. Пакеты ROS/URDF **установлены** и
  прописаны в `Packages/manifest.json`, их сборки лежат в `Library/PackageCache`.
- `Assembly-CSharp.csproj` действительно ссылается на `Unity.Robotics.ROSTCPConnector` —
  значит `ROSBridge.cs` в редакторе компилируется. Явных синтаксических ошибок в коде нет.
- Вывод: скорее всего это **stale-кэш Burst/Bee после добавления пакетов**, либо ошибка
  видна только при сборке под конкретную платформу (Dedicated Server).

**Как чинить (по порядку, от дешёвого к дорогому):**
1. Открыть **Console** в Unity, выключить `Collapse`, промотать к **самой первой красной**
   ошибке (`CS####`). Cecil-ошибка обычно ниже по списку — чините первопричину сверху.
   Если Console **чист** — переходите к п.2.
2. **Очистить кэши компиляции** (безопасно, Unity пересоберёт):
   - Закрыть Unity.
   - Удалить папки: `Library/Bee`, `Library/BurstCache`, `Library/ScriptAssemblies`, `Temp`.
   - Открыть проект заново, дождаться полной переимпортации/компиляции.
3. Если ошибка **только при сборке билда**: собирать **Standalone Linux**, а НЕ *Dedicated
   Server* (см. прошлый отчёт `WORK_REPORT.md` §4 — на Dedicated Server ещё и `grpc`-`.so`
   не подхватывается). Меню *File → Build Settings → Target Platform = Linux*, снять галку
   *Dedicated Server* (использовать обычный Player), снять *Development Build*.
4. Если и это не помогло — прислать **первую** ошибку из Console: без неё Cecil-сообщение
   диагностируется только гаданием.

> Кэш `Library` я **намеренно не удалял** сам: это тяжёлая операция, и её лучше делать при
> закрытом редакторе с вашего ведома. Команда (из корня проекта, Unity закрыт):
> ```bash
> rm -rf Library/Bee Library/BurstCache Library/ScriptAssemblies Temp
> ```

## 2. Переработка наград и штрафов

### 2.1. Что нового в `RobotBrain.cs`
Всё **аддитивно**: старая механика shaping'а (potential-based) сохранена, накрутить награду
кружением по-прежнему нельзя.

- **Лимит времени — 90 с (1.5 минуты).** Поле `episodeTimeLimit` по умолчанию = `90f`,
  ключ `episode_time_limit: 90.0` в yaml.
- **Штрафы за фактические столкновения** (новое):
  - `wall_collision_penalty` — за касание **стены**;
  - `obstacle_collision_penalty` — за касание **преграды/коробки**.
  - Реализованы через `OnCollisionEnter/Stay`. **Пол в сцене на слое Obstacle**, поэтому
    контакты с почти вертикальной нормалью (пол/потолок) **отфильтрованы** — робот НЕ
    штрафуется за езду по полу. Засчитываются только контакты, чей слой попал в маску
    `Wall Collision Mask` / `Obstacle Collision Mask`.
- **Штрафы за направление движения** (новое/расширено): `spin_penalty` (|руль|),
  `forward_penalty` (газ вперёд), `reverse_penalty` (газ назад). По умолчанию все `0`.
- **Штраф за каждый шаг** — уже был (`time_penalty`), оставлен.
- **`obstacle_penalty`** переосмыслен как штраф за *близость* по ИК (в отличие от нового
  штрафа за *касание*).

### 2.2. `config.yaml` — чистый YAML без комментариев
- Все награды/штрафы — в `environment_parameters`, **без комментариев**, блоки разделены
  **пустой строкой** (целевые бонусы / терминальные штрафы / пошаговые штрафы / столкновения
  / направление / служебное / эпизод / движение / спавн / DR).
- Проверено парсером PyYAML: файл валиден, все 27 значений числовые.

### 2.3. Отдельный файл-справочник
- **`REWARDS_REFERENCE.md`** (новый) — за что отвечает **каждый** параметр награды/штрафа,
  куда его крутить и быстрые рецепты. Именно этот файл заменяет удалённые из yaml
  комментарии.
- **`CONFIG_REFERENCE.md`** — обновлён (новые ключи, лимит 90 с, пометка про чистый yaml).

## 3. Статистика попыток — новый компонент `RobotStats.cs`

Новый скрипт `Assets/Scripts/RobotStats.cs`. `RobotBrain` в конце **каждой** попытки
(единая точка `EndAttempt()`) отдаёт в него результат. Куда уходят данные:

1. **TensorBoard** (через `Academy.StatsRecorder`) — видно во время обучения:
   - `Attempt/Success`, `Attempt/Timeout`, `Attempt/Fall` — доли исходов;
   - `Attempt/DurationSec` — **средняя длительность попытки**;
   - `Attempt/Grabs` — **сколько раз реально взял мяч**;
   - `Attempt/TotalReward` — награда за попытку;
   - `Rewards/Approach`, `Rewards/ActionPenalty`, `Rewards/TimePenalty`,
     `Rewards/ObstacleProx`, `Rewards/WallCollision`, `Rewards/ObstCollision`,
     `Rewards/DirectionPen` — **распределение компонентов награды**.
2. **Экранная сводка (OnGUI)** — для визуального/ручного запуска: число попыток, % успеха,
   всего захватов, среднее время, средняя/последняя награда.
3. **CSV (опция)** — включается галкой `writeCsv`: по строке на попытку в папку `StatsCSV/`
   рядом с проектом (для анализа распределения наград/длительностей в Excel/pandas).

## 4. Рандомизация поля при визуальном запуске (`--interact`)

Раньше рандомизация арены шла **только при обучении** (`isTraining`), поэтому в ручном/
визуальном запуске поле было статичным.

- Добавлен флаг инспектора `RobotBrain.randomizeArenaInPlay` (по умолчанию **вкл**).
- Введено разделение: `ApplyArenaRandomization` (арена + точка спавна мяча — работает и в
  ручном запуске) против `ApplyDR` (шумы сенсоров, задержки, масса — только обучение, чтобы
  ручное вождение оставалось «чистым»).
- Теперь при каждом старте эпизода в Play-режиме коробки/свет/трение и позиция мяча
  перетасовываются.

> Замечание: значения из `config.yaml` в ручном запуске **не читаются** (нет тренера),
> поэтому флаг сделан полем инспектора, а не ключом yaml.

## 5. Несколько арен с роботами (prefab) — пошагово в редакторе

Вы выбрали **настоящий `.prefab`-ассет**. Из WSL создать его нельзя (нужен редактор),
поэтому вот точная инструкция. Код к этому уже готов: `EnvironmentRandomizer` сделан
**prefab-дружелюбным** — добавлено поле `Arena Origin`. Если его задать (перетащить корень
арены), каждая копия рандомизируется вокруг **своего** центра, а не мирового нуля.

**Шаг 1. Собрать арену под один корень.**
1. В Hierarchy создайте пустой объект `Arena` в позиции `(0,0,0)`.
2. Перетащите внутрь него **всё, что относится к одной арене**: робота (с `RobotBrain`,
   `TrackController` и т.д.), `Target_ball`, `Environment` (пол, стены, коробки, свет,
   `EnvironmentRandomizer`). Всё это станет детьми `Arena`.
3. Убедитесь, что ссылки в компонентах указывают на объекты **внутри** этой же арены
   (они и так внутренние). В `EnvironmentRandomizer` заполните новое поле **`Arena Origin`**
   = объект `Arena`.

**Шаг 2. Сделать prefab.**
1. Перетащите `Arena` из Hierarchy в папку `Assets/Prefabs` (создайте её). Получится
   `Arena.prefab`, а объект в сцене станет его экземпляром (синий).

**Шаг 3. Разложить копии.**
- Вручную: перетащите `Arena.prefab` в сцену несколько раз, расставив по сетке со сдвигом
  (например, шаг 6 м по X/Z): `(0,0,0)`, `(6,0,0)`, `(12,0,0)`, `(0,0,6)` и т.д. Копии не
  должны пересекаться. Благодаря `Arena Origin` рандомизация в каждой копии — своя.

**Шаг 4. Проверить ML-Agents.**
- У всех роботов **одинаковый Behavior Name** (`GFSX_Brain`) — так и должно быть: все копии
  учат одну общую политику. В `config.yaml` менять ничего не надо.
- `Decision Requester` и прочие компоненты копируются вместе с prefab автоматически.
- Больше арен = быстрее сбор опыта на одной машине. Начните с 4–9 копий.

> Альтернатива без ручной раскладки — скрипт-клонер в `Awake`, но вы выбрали настоящий
> prefab, поэтому клонер не добавлял.

## 6. Что нужно сделать в редакторе (чек-лист)

> Это правки **C#-кода**, поэтому билд нужно **пересобрать один раз**. После — значения в
> `environment_parameters` меняются без пересборки.

1. **Компиляция.** Открыть проект, дождаться компиляции, убедиться, что Console **без
   красных ошибок** (если есть — см. §1).
2. **Статистика.** На объект робота добавить компонент **`RobotStats`**. В компоненте
   `RobotBrain` в поле **`Stats`** перетащить этот `RobotStats`. (Без назначения статистика
   просто не собирается — код это переживает.)
3. **Слои столкновений (для штрафов за стены/коробки).**
   - Коробки уже на слое `Obstacle`. Если маски в `RobotBrain` не трогать — и стены, и
     коробки будут штрафоваться **одним** параметром `obstacle_collision_penalty` (пол
     отфильтрован по нормали).
   - Чтобы штрафы за **стены** и **коробки** были **раздельными**: создайте слой,
     например `Wall` (*Layers → Edit Layers*), повесьте его на объекты-стены, затем в
     `RobotBrain`: `Wall Collision Mask = Wall`, `Obstacle Collision Mask = Obstacle`.
4. **Рандомизация в Play** уже включена флагом `Randomize Arena In Play` (по умолчанию вкл).
   Если хотите статичное поле для отладки — снимите галку.
5. **Prefab с несколькими аренами** — см. §5.
6. **ROSBridge** (из вашего примечания «не успел добавить к роботу»): повесьте компонент
   `ROSBridge` на робота и в `RobotBrain` заполните поле `Ros Bridge`. Он сам отключается
   на обучении и публикует топики только на инференсе — на обучение не влияет.
7. **Пересобрать билд**: Standalone **Linux** (не Dedicated Server), без Development Build.
8. Запустить прогон с **новым** `--run-id` (не перетирать старые результаты в `results/`).

## 8. Почему это «ничего не сломало»

- Новые штрафы (столкновения, направление) по умолчанию либо `0`, либо срабатывают только
  по назначенной маске + горизонтальной нормали → **на текущее обучение не влияют**, пока
  вы их не включите/не назначите слои.
- Поле `stats` и `rosBridge` необязательны: `null` обрабатывается.
- `RobotStats` только читает и пишет метрики — на физику и политику не влияет.
- `EnvironmentRandomizer.Arena Origin` при пустом значении = **прежнее** мировое поведение.
- `config.yaml`: изменились только числа/структура; секция `behaviors` (PPO) не тронута;
  файл проверен парсером.
- **Единственное умышленное изменение поведения по ТЗ** — лимит эпизода 20 → 90 с.
