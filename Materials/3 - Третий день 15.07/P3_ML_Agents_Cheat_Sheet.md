## 🧐 Теория: Зачем, Почему и Как работает обучение агента?

Перед тем как настраивать инспектор и писать код, важно понять принципы работы обучения с подкреплением (Reinforcement Learning) для этой задачи:

| Параметр / Шаг | ❓ ЗАЧЕМ это нужно | 🧠 ПОЧЕМУ именно так реализовано | 🛠 КАК это настроить |
| :--- | :--- | :--- | :--- |
| **LSTM-память (в config.yaml)** | Реальный робот имеет слепые зоны (например, мяч скрывается под бампером перед захватом) и задержки сигналов. | Без памяти робот будет «забывать» мяч сразу, как только тот пропадет с камеры, и начнет крутиться на месте. Память позволяет доезжать до цели вслепую. | В `config.yaml` мы подключаем блок `memory` (с длиной истории `sequence_length: 64`). |
| **Stacked Vectors = 4** | Нейросеть должна понимать динамику: робот ускоряется или тормозит, мяч катится к нам или от нас. | Передача только текущих координат дает статичную картинку. Стек из 4 кадров истории дает сети понимание векторов скоростей без математики производных. | В инспекторе робота в компоненте `Behavior Parameters` выставляем `Stacked Vectors = 4`. |
| **Decision Period = 5** | Делать запросы к ИИ-модели каждый физический тик (50 Гц) слишком дорого для процессора Raspberry Pi. | Мотор робота физически не успевает отработать смену команд за 20 мс. Период 5 означает смену управляющих сигналов каждые 100 мс, что делает движение более плавным. | В компоненте `Decision Requester` выставляем `Decision Period = 5`. |
| **Эгоцентрические смещения (egoDisplacement)** | LSTM-памяти для построения внутренней карты пути нужно знать свое перемещение относительно старта. | Без одометрии (смещения и направления) память не сможет связать пропажу мяча с собственным движением робота вперед. | В C# мы считываем разность текущих координат с точкой старта и отправляем в `CollectObservations`. |

## 🗂 Пошаговый чек-лист настройки в Unity

### Шаг 1: Настройка компонентов ML-Agents на роботе
1. Добавьте компонент **Behavior Parameters** на корневой объект робота:
   * **Behavior Name**: Имя «мозга» вашего робота на ваш выбор (например, `GFSX_Brain`).
   * **Vector Observation ➔ Space Size**: `15` (мы собираем ровно 15 параметров).
   * **Vector Observation ➔ Stacked Vectors**: `4`.
   * **Actions ➔ Continuous Actions**: `3` (газ, руль, поворот камеры).
   * **Actions ➔ Discrete Branches**: `1` (размерность ветки `Branch 0 = 3` — для управления клешней: 0 = NOOP, 1 = Close, 2 = Open).
2. Добавьте компонент **Decision Requester** и установите **Decision Period**: `5`.

### Шаг 2: Создание скрипта SimulatedYoloCamera.cs
Создайте C# скрипт **`SimulatedYoloCamera`** и навесьте его на камеру робота.

*   **Принцип работы (Проекция YOLO)**:
    Реальный робот ищет мяч с помощью бортовой камеры и нейросети YOLO, которая возвращает координаты рамки (Bounding Box) на изображении. В симуляторе нам не нужно тратить ресурсы процессора на рендеринг картинки. 
    Мы можем спроецировать 3D-координаты мяча в 2D-координаты экрана с помощью метода камеры `WorldToViewportPoint(targetBall.position)`.
*   **Логика расчетов**:
    Viewport-координаты в Unity изменяются в диапазоне от 0 до 1. 
    1. Переведите горизонтальную координату проекции ($x$) так, чтобы значение отражало отклонение относительно центра кадра (например, от -1 слева до +1 справа). Это будет относительный угол до мяча.
    2. Разделите физическое расстояние до мяча на максимальную дистанцию видимости камеры (например, 2 метра) для получения нормализованного значения от 0 (вплотную) до 1 (далеко).
*   **Проверка видимости**:
    Мяч считается видимым, если он попал в поле зрения камеры (hFOV = 40°), находится в пределах максимальной дальности и между камерой и мячом нет стен (проверяется с помощью `Physics.Raycast`).

## 💻 Шаг 3: Написание кода ИИ-агента (RobotBrain.cs)
Создайте C# скрипт **`RobotBrain`**, наследующий класс **`Agent`** (требуется `using Unity.MLAgents;`). Навесьте его на робота.

### 1. Метод сбора наблюдений: `CollectObservations`
Передайте ваши параметры в датчик наблюдений (`VectorSensor sensor`). Порядок добавления параметров должен строго совпадать с тем, что вы настроите в коде:

*Рекомендуемая базовая структура вектора наблюдений (15 значений):*
1.  Нормализованное расстояние с УЗ-дальномера.
2.  Левый ИК-датчик препятствий (0/1).
3.  Правый ИК-датчик препятствий (0/1).
4.  ИК-датчик клешни (0/1).
5.  Относительный горизонтальный угол до мяча по камере (0, если мяч не виден).
6.  Нормализованное расстояние до мяча по камере (1, если мяч не виден).
7.  Последнее известное направление на мяч (после утери из кадра).
8.  Флаг видимости мяча (0.0 или 1.0).
9.  Текущий поворот сервопривода камеры.
10. Статус захвата мяча клешнёй (`hasBall` ➔ 0.0 или 1.0).
11. Относительное смещение робота по оси X от точки старта.
12. Относительное смещение робота по оси Z от точки старта.
13. Нормализованный угол направления взгляда робота (Heading).
14. Текущая скорость движения робота от Rigidbody.
15. Время, прошедшее с момента последней детекции мяча.

### 2. Метод приема действий: `OnActionReceived`
Считайте сигналы непрерывных действий:
*   `actions.ContinuousActions[0]` ➔ линейное движение (передайте в `TrackController`).
*   `actions.ContinuousActions[1]` ➔ угловое вращение (передайте в `TrackController`).
*   `actions.ContinuousActions[2]` ➔ сигнал поворота камеры.

Считайте сигнал дискретного действия:
*   `actions.DiscreteActions[0]` ➔ команда клешне (0 = стоять, 1 = закрыть, 2 = открыть).

## 💻 Шаг 4: Разработка функции наград (Rewards)
В конце метода `OnActionReceived` вызовите расчет наград. Рекомендуемые принципы поощрений:

1.  **Награда за сближение (Distance Delta)**:
    Сравнивайте расстояние до мяча на предыдущем и текущем шаге. Начисляйте небольшую награду за приближение к цели. Сближение вблизи мяча рекомендуется поощрять сильнее, чем вдали.
2.  **Штраф за резкость управления (Action Rate Penalty)**:
    Чтобы робот не дергался и двигался плавно, штрафуйте его за слишком резкое изменение сигналов газа и руля между шагами.
3.  **Бонус за центрирование**:
    Поощряйте разворот робота передом к мячу перед попыткой схватить его.
4.  **Штраф за датчики расстояния**:
    Начисляйте штраф при критическом сближении со стенами по датчикам, чтобы научить агента маневрировать.
5.  **Терминальные условия**:
    *   **Успешный финал**: Мяч захвачен и удерживается ➔ начислите крупный терминальный бонус (например, `+5.0`) и завершите эпизод через `EndEpisode()`.
    *   **Падение**: Если робот вылетел за пределы арены ➔ начислите штраф и завершите эпизод.

## 💻 Шаг 5: Конфигурационный файл `config.yaml`
Создайте текстовый файл `config.yaml` в корне вашего проекта и настройте параметры PPO-обучения.

> [!WARNING]
> **Важно**: Внутри файла `config.yaml` не должно быть русских символов и комментариев на кириллице! Это вызовет синтаксическую ошибку при парсинге файла библиотекой PyYAML в Python. Ниже представлен чистый код конфигурации без комментариев.

```yaml
behaviors:
  GFSX_Brain:
    trainer_type: ppo
    hyperparameters:
      batch_size: 2048
      buffer_size: 40960
      learning_rate: 0.00025
      beta: 0.005
      epsilon: 0.2
      lambd: 0.95
      num_epoch: 3
      learning_rate_schedule: linear
    network_settings:
      normalize: true
      hidden_units: 256
      num_layers: 2
      memory:
        sequence_length: 64
        memory_size: 256
    reward_signals:
      extrinsic:
        gamma: 0.99
        strength: 1.0
    keep_checkpoints: 5
    max_steps: 5000000
    time_horizon: 1000
    summary_freq: 20000
```

### 🔍 Расшифровка параметров конфигурации:

*   **`trainer_type: ppo`**: Алгоритм обучения Proximal Policy Optimization (RL).
*   **`batch_size: 2048`**: Объем данных опыта (в шагах) для одного обновления нейросети.
*   **`buffer_size: 40960`**: Общий объем собранного опыта (шагов) перед тем, как запустить обновление весов.
*   **`learning_rate: 0.00025`**: Скорость обучения (шаг градиентного спуска).
*   **`beta: 0.005`**: Энтропия (отвечает за «любопытство» и случайность действий). Помогает роботу исследовать арену.
*   **`normalize: true`**: Приведение вектора наблюдений к единому масштабу для корректного сопоставления разнородных данных с датчиков.
*   **`hidden_units: 256` и `num_layers: 2`**: Размеры нейросети (2 скрытых слоя по 256 нейронов).
*   **`memory` (LSTM)**: Подключает рекуррентную память, чтобы робот мог учитывать предысторию событий и компенсировать задержки.
*   **`gamma: 0.99`**: Коэффициент дисконтирования наград (насколько робот ценит будущую цель по сравнению с мгновенными бонусами).
*   **`max_steps: 5000000`**: Общее количество шагов симуляции, после которого обучение остановится. Для полноценного обучения рекомендуется выставлять **от 5 до 10 млн шагов**.
*   **`summary_freq: 20000`**: Частота записи данных для TensorBoard.

## 🔌 Шаг 8: Мост интеграции с реальным роботом (Создание ROSBridge.cs)

После того как вы обучили нейросеть, её необходимо запустить на выполнение (инференс) и связать с физическим роботом GFS-X. Для этого вам предстоит самостоятельно создать C# скрипт-мост **`ROSBridge.cs`**.

### ❓ ЗАЧЕМ он нужен?
Скрипт транслирует управляющие решения, принятые обученной нейросетью внутри Unity, в стандартные топики ROS (Robot Operating System) на реальный бортовой компьютер Raspberry Pi робота по TCP-каналу (через ROS TCP Connector).

### 🧠 ПОЧЕМУ в нем реализуется сглаживание (EMA)?
Нейросети принимают решения с высокой частотой и часто «вибрируют» (колеблются между соседними значениями скорости).
*   **Опасность**: В реальности этот дребезг приводит к тому, что моторы робота начинают гудеть, греться, а шестерни редуктора быстро изнашиваются и выходят из строя.
*   **Решение — EMA-фильтрация (Exponential Moving Average)**:
    В скрипте применяется формула экспоненциального сглаживания команд перед отправкой в ROS:
    `smoothGas = emaAlpha * gas + (1f - emaAlpha) * smoothGas`
    Коэффициент `emaAlpha = 0.8f` отсекает высокочастотную тряску сети, сохраняя при этом мгновенную реакцию робота на резкие повороты.
*   **Защита от дрейфа (Hard Stop)**:
    Если пришла чистая команда остановки ($gas = 0, steering = 0$), сглаженные скорости сбрасываются в $0$ мгновенно, без затухания, чтобы избежать паразитного дрейфа робота из-за "ошметков" фильтра.

### 📋 Структурный шаблон для написания ROSBridge.cs:
Создайте новый скрипт `ROSBridge.cs` и реализуйте в нем следующую логику:

```csharp
using UnityEngine;
using Unity.Robotics.ROSTCPConnector;
using RosMessageTypes.Geometry; // Требуется для TwistMsg
using RosMessageTypes.Std;      // Требуется для Int32Msg и Float32Msg

public class ROSBridge : MonoBehaviour
{
    public string topicName = "/cmd_vel";
    public float maxLinearSpeed = 0.5f;   // Линейный лимит реального робота (м/с)
    public float maxAngularSpeed = 1.0f;  // Угловой лимит реального робота (рад/с)

    [Range(0.1f, 1f)]
    public float emaAlpha = 0.8f;         // Коэффициент сглаживания (0.8 = высокая отзывчивость)

    private ROSConnection ros;
    private float smoothGas = 0f;
    private float smoothSteering = 0f;

    void Start()
    {
        // Получаем экземпляр ROS-подключения
        ros = ROSConnection.GetOrCreateInstance();
        
        // Регистрируем топики для публикации
        ros.RegisterPublisher<TwistMsg>(topicName);
        ros.RegisterPublisher<Int32Msg>("/cmd_gripper");
        ros.RegisterPublisher<Float32Msg>("/cmd_camera_pan");
    }

    // Метод отправки сглаженных скоростей в /cmd_vel
    public void PublishCommand(float gas, float steering)
    {
        if (Mathf.Approximately(gas, 0f) && Mathf.Approximately(steering, 0f))
        {
            smoothGas = 0f;
            smoothSteering = 0f;
        }
        else
        {
            smoothGas = emaAlpha * gas + (1f - emaAlpha) * smoothGas;
            smoothSteering = emaAlpha * steering + (1f - emaAlpha) * smoothSteering;
        }

        TwistMsg cmd = new TwistMsg();
        cmd.linear.x = smoothGas * maxLinearSpeed;
        cmd.angular.z = smoothSteering * maxAngularSpeed;

        ros.Publish(topicName, cmd);
    }

    // Метод отправки команд манипулятора в /cmd_gripper
    public void PublishGripperCmd(int cmd)
    {
        Int32Msg msg = new Int32Msg();
        msg.data = cmd;
        ros.Publish("/cmd_gripper", msg);
    }

    // Метод отправки угла камеры в /cmd_camera_pan
    public void PublishCameraCmd(float yaw)
    {
        Float32Msg msg = new Float32Msg(yaw);
        ros.Publish("/cmd_camera_pan", msg);
    }
}
```

### 🛠 КАК мост связывается с вашим ИИ-агентом?
1. Повесьте ваш скрипт `ROSBridge.cs` на робота в Unity.
2. Внутри метода `OnActionReceived()` вашего скрипта агента `RobotBrain.cs` добавьте вызовы методов моста, которые будут пересылать сигналы только во время инференса на реальном роботе (`isTraining == false`):
   ```csharp
   // Внутри OnActionReceived() при управлении:
   if (!isTraining && rosBridge != null)
   {
       rosBridge.PublishCommand(gas, steering);
       rosBridge.PublishCameraCmd(cameraYawInput);
   }

   // При срабатывании захвата мяча:
   if (!isTraining && rosBridge != null)
   {
       rosBridge.PublishGripperCmd(2); // Отправить команду закрытия в ROS
   }
   ```

### ⚠️ Задачи на доработку и кастомизацию для студентов:
> [!WARNING]
> **Не копируйте шаблон вслепую!** Простой перенос кода из шпаргалки не гарантирует стабильную работу на реальном полигоне. Каждая команда обязана кастомизировать и доработать скрипт `ROSBridge.cs` под следующие требования:

1.  **Реализуйте защитный таймер (Watchdog / Fail-safe)**:
    *   *Проблема*: Если во время движения робота Wi-Fi соединение оборвется или Unity-приложение зависнет, ROS-нода на Raspberry Pi продолжит выполнять последнюю пришедшую скорость. Робот на полной скорости врежется в стену или собьет оборудование.
    *   *Задача*: Напишите в `ROSBridge.cs` логику безопасности. Если скрипт не получает вызовов `PublishCommand` от ИИ-агента в течение последних `0.5` секунд (например, через замер `Time.time` последнего вызова в `Update`), он должен автоматически отправить экстренную команду полной остановки (`linear.x = 0, angular.z = 0`) и вывести предупреждение в консоль.
