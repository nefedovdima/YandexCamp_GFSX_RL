# 🧠 Логика проекта GFS-X (словами + числа)

Единое описание того, как устроен симулятор робота-скаута GFS-X: архитектура,
наблюдения, действия, жизненный цикл эпизода, награды, Domain Randomization и все
ключевые числа. Значения соответствуют коду и `config.yaml` на момент написания.

> Связанные файлы: `REWARDS_REFERENCE.md` (детальная таблица наград),
> `CONFIG_REFERENCE.md` (гиперпараметры PPO), `WORK_REPORT_2.md` (что и зачем менялось).

---

## 1. Задача

Гусеничный робот на арене 3×3 м должен **найти мяч, подъехать и захватить его клешнёй**,
удерживая ~1 секунду. Обучение — Reinforcement Learning (PPO + LSTM) через Unity ML-Agents.
Цель — политика, которая переносится на реальный XiaoR GFS-X (Raspberry Pi + ROS).

---

## 2. Архитектура (скрипты и их роли)

| Скрипт | Роль |
| :--- | :--- |
| `RobotBrain.cs` | Мозг-агент (наследник `Agent`). Собирает 15 наблюдений, принимает действия, считает награды, ведёт эпизод и Domain Randomization. |
| `TrackController.cs` | Дифференциальный привод: команды газ/руль → скорости гусениц → PWM с мёртвой зоной → перемещение через `MovePosition`/`MoveRotation`. |
| `VirtualSensors.cs` | Виртуальные датчики: УЗ-дальномер (веер лучей), боковые ИК, ИК клешни (фронтальный конус). |
| `SimulatedYoloCamera.cs` | Имитация YOLO-камеры через `WorldToViewportPoint`: угол/дистанция/видимость мяча. Рисует пирамиду FOV. |
| `GripperController.cs` | Логический захват: при обнаружении мяча делает его kinematic, отключает коллайдер, крепит к `HoldPoint`. |
| `EnvironmentRandomizer.cs` | Рандомизация арены: коробки, свет, трение, точка спавна мяча (в т.ч. скрытая). |
| `RobotStats.cs` | Сбор статистики попыток → TensorBoard (`Attempt/*`, `Rewards/*`) + экранная сводка + CSV. |
| `ROSBridge.cs` | Мост в ROS (только инференс): публикует `/cmd_vel`, `/cmd_gripper`, `/cmd_camera_pan` с EMA-сглаживанием и watchdog. |

**Такты:** физика 50 Гц (`fixedDeltaTime = 0.02 с`); решение сети — раз в 5 физических
тиков (`Decision Period = 5` → смена команд каждые 100 мс); стек из 4 кадров наблюдений
(`Stacked Vectors = 4`).

| 6 | Дистанция до мяча по камере (1, если не виден) | 0…1 | ✅ ~из bbox |

| 11 | Смещение X от старта (одометрия) + шум/дрейф | −1…1 | ⚠️ только счисление |

+ накопительный дрейф + шум (см. §7), чтобы политика не полагалась на точные координаты.

## 4. Действия

**Непрерывные (3):**
- `[0]` газ (−1 назад … +1 вперёд)
- `[1]` руль (−1 влево … +1 вправо)
- `[2]` скорость поворота серво камеры (−1…+1), интегрируется в угол ±`camServoMaxAngle`

**Дискретное (1 ветка, 3 значения):**
- `0` = стоять, `1` = закрыть клешню (захват), `2` = открыть

При обучении команды проходят через **очередь задержки** (Latency Queue) 8–13 тиков
(160–260 мс) — имитация пинга Wi-Fi/ROS.

## 5. Жизненный цикл эпизода

**`OnEpisodeBegin` (по порядку):**
1. Подтянуть параметры из `config.yaml` (`ApplyTunables`).
2. Отпустить мяч, если держали.
3. Телепортировать робота на старт, обнулить скорости.
4. Рандомизировать арену (коробки/свет/трение) — при обучении всегда, в Play — если включён `randomizeArenaInPlay`.
5. Сбросить камеру в стартовую позу.
6. Спавн мяча: с вероятностью `hidden_spawn_fraction` — **скрытая** точка (за стеной/вне кадра), иначе любая валидная.
7. Сбросить счётчики (столкновения, статистика, дрейф одометрии).
8. Задать базовый потенциал Ф(s) для shaping.
9. При обучении: рандомизация массы/моторов, заполнить очередь задержки.

**Условия завершения эпизода:**
| Исход | Условие | Награда |
| :--- | :--- | :--- |
| **Успех** | мяч удержан `hold_ticks_required` (50 тиков ≈ 1 с) | `+grab_reward` |
| **Таймаут** | истёк `episode_time_limit` без захвата | `+timeout_penalty` |
| **Падение** | Y ниже `fallY` (−1 м) | `+fall_penalty` — ⚠️ недостижимо, если в Rigidbody заморожен Y |

Все выходы идут через `EndAttempt(outcome)` → отдают метрики в `RobotStats` → `EndEpisode()`.

## 6. Функция награды (все слагаемые)

Награда за шаг = сумма слагаемых. Числа — текущие значения из `config.yaml`.

### 6.1. Ядро — potential-based shaping
`r = shaping_gamma·Ф(s') − Ф(s)`, где потенциал
`Ф = approach_reward·DistPot(dist) + centering_reward·(1 − |пеленг|/180)`.
- Теорема Ng et al. (1999): не меняет оптимальную политику и **не даёт накрутить награду
  кружением** — сумма по любому замкнутому маршруту ≈ 0. Платит только реальный прогресс к мячу.
- `approach_reward = 1.0`, `centering_reward = 0.5`, `shaping_gamma = 0.99` (обязан совпадать с `gamma` PPO).
- ⚠️ При `gamma<1` shaping «подтекает»: далеко Ф отрицателен → крохотный плюс за шаг. Гасится `time_penalty` (см. 6.4).

### 6.2. Целевые бонусы
| Слагаемое | Значение | Когда |
| :--- | :--- | :--- |
| `grab_reward` | +20.0 | терминально, за успешный захват+удержание |
| `hold_tick_reward` | +0.05/тик | пока мяч удерживается |
| `gripper_ir_reward` | +0.03/тик | мяч в луче клешни, но ещё не схвачен («последний метр») |
| `search_find_reward` | +0.5 разово | первое обнаружение мяча в эпизоде (учит искать) |

### 6.3. Штрафы за плавность/направление
| Слагаемое | Значение | За что |
| :--- | :--- | :--- |
| `action_penalty` | 0.02 | резкость: \|Δгаз\|+\|Δруль\|+\|Δкамера\| |
| `gas_reversal_penalty` | 0.005 | смена знака газа (дёрганье вперёд-назад) |
| `cam_move_penalty` | 0.002 | \|поворот камеры\| всегда (плавность серво) |
| `cam_visible_move_penalty` | 0.03 | движение камеры, пока мяч виден (не терять цель) |
| `spin_penalty` | 0.0 | \|руль\| (выкл: при узком FOV робот обязан вертеться) |
| `forward_penalty` | 0.0 | газ вперёд (обычно не нужен) |
| `reverse_penalty` | 0.01 | газ назад (клешня/камера спереди) |
| `near_ball_speed_penalty` | 0.05 | скорость+\|руль\| вблизи мяча, нелинейно ∝prox² (аккуратный подъезд) |

### 6.4. Ургентность и препятствия
| Слагаемое | Значение | За что |
| :--- | :--- | :--- |
| `time_penalty` | 0.02/тик | каждый шаг (срочность + гасит протечку shaping) |
| `obstacle_penalty` | 0.015 | близость к преграде по боковым ИК (сближение) |
| `wall_collision_penalty` | 0.06 | фактическое касание стены (фильтр пола по нормали) |
| `obstacle_collision_penalty` | 0.03 | фактическое касание коробки/преграды |

### 6.5. Терминальные штрафы
`fall_penalty = −1.0` (недостижим при заморозке Y), `timeout_penalty = −0.3`.

> Все слагаемые «около мяча» и shaping считаются по **истинной геометрии** (реальная позиция
> мяча). Это законно: награда существует только при обучении, политика её не видит — она
> читает лишь 15 наблюдений. Привилегированная инфа в награде ≠ утечка симулятора.

## 7. Domain Randomization (мастер-выключатель `dr_enabled = 1.0`)

Только при обучении. Учит политику терпеть разброс реального мира.

| Что рандомизируется | Диапазон/значение |
| :--- | :--- |
| Масса робота | `mass_min…mass_max` = 1.0…4.0 кг (база 2.5) |
| Скорость привода (moveSpeed) | 0.3…0.7 м/с |
| Скорость поворота (turnSpeed) | 80…160 град/с |
| Инерция разгона (maxPwmStep) | 8…25 |
| Масса мяча | ×0.5…×2 |
| Размер мяча | ×0.8…×1.2 |
| Шум УЗ-дальномера | `us_noise` = ±0.05 |
| Флип боковых ИК | шанс 0.02/тик |
| Потери YOLO (dropout) | при рывке (>30 град/с) шанс `yolo_dropout_chance`=0.15 на 5–15 тиков; фоновый 0.01 на 3–8 тиков |
| Задержка команд (latency) | `latency_min…max` = 8…13 тиков (160–260 мс) |
| Одометрия: дрейф | `odom_drift` = 0.03 м (std за ~1 с, random walk ∝√t) |
| Одометрия: шум | `odom_noise` = 0.01 м/тик |
| Одометрия: масштаб | ±5% на эпизод |
| Одометрия: курс, дрейф | 3 град (std за ~1 с) |
| Арена | позиции/повороты коробок, свет, трение пола |

## 8. Сложность и поиск

| Параметр | Значение | Смысл |
| :--- | :--- | :--- |
| `hidden_spawn_fraction` | 1.0 | доля эпизодов, где мяч **не виден со старта** (за стеной/вне кадра). 1.0 = всегда вслепую → максимум поиска |
| `ball_spawn_radius` | 0.0 | 0 = мяч по всей арене (без облегчения); >0 = не дальше R (только для диагностики) |
| `box_count_min/max` | 4 / 15 | сколько коробок-преград за эпизод (`max:0` = без преград) |
| `ball_spawn_jitter` | 0.15 | сдвиг мяча, если случайная точка не нашлась |

**Curriculum сознательно НЕ используется** — учим сразу на полной сложности.

## 11. Принципы, которые нельзя нарушать

1. **Вектор наблюдений — ровно 15.** Изменение размера ломает совместимость с обученной
   моделью (нужен переучёт + правка `Space Size`).
2. **`shaping_gamma` = `gamma` PPO.** Иначе shaping перестаёт быть policy-invariant.
3. **Мёртвая зона моторов — в физике `TrackController`, а не в награде.** Штраф за «малое
   руление» не добавлять — он толкает к bang-bang и возвращает дёрганье.
4. **Аккуратность у мяча — штрафом, а не наградой за медленность** (награду фармят зависанием).
5. **Одометрия (obs 11–13) — реалистично зашумлена.** На реальном роботе туда должно идти
   счисление по командам (`cmd_vel`-интеграция), иначе — нули/мусор.
6. **ROSBridge активен только на инференсе**; на обучении отключается сам.

## 12. Метрики (`RobotStats`, если подключён)

`Attempt/Success` (доля успехов), `Attempt/Grabs` (захватов/эпизод),
`Attempt/DurationSec` (ср. время попытки), `Attempt/WallHits` / `Attempt/ObstacleHits`
(врезаний/эпизод), `Attempt/Timeout` / `Attempt/Fall`, а также `Rewards/*` — распределение
компонентов награды. Косвенный индикатор без RobotStats — `Environment/Episode Length`
(падает = чаще/быстрее берёт мяч).
